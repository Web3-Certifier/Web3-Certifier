# Fix: Gas Fees Too Low Causing Transaction Failures

## Problem
In `wagmiWrite.ts`, gas fees are hardcoded with very low values:
- `maxFeePerGas: 25.001e9` (25.001 Gwei)
- `maxPriorityFeePerGas: 1e8` (0.1 Gwei)

These values are insufficient for current network conditions (2026), causing transactions to fail as they don't get included in blocks.

## Root Cause
Lines 85-87 in `packages/nextjs/hooks/wagmi/wagmiWrite.ts` hardcode EIP-1559 gas fee parameters. These static values don't adapt to network congestion and are now far below market rates.

## Solution
Replace hardcoded gas fees with dynamic network-based estimation:

1. Use `walletClient.estimateFeesPerGas()` to get current network gas prices
2. Remove the hardcoded `maxFeePerGas` and `maxPriorityFeePerGas` values
3. Let the transaction use dynamically estimated fees
4. Keep `type: 2` for EIP-1559 transactions
5. Add error handling fallback for networks that don't support fee estimation

## Implementation Steps

### File to Modify
`packages/nextjs/hooks/wagmi/wagmiWrite.ts`

### Changes
1. In the `writeWithParams()` function (lines 76-93), replace the hardcoded gas parameters with dynamically estimated ones
2. Use `walletClient.estimateFeesPerGas()` or let wagmi handle it automatically by omitting these parameters
3. Since `wagmiSendTransaction.sendTransactionAsync` can handle gas estimation automatically, we can simply remove the hardcoded values and let the wallet/client handle it

### Code Change Details
Current (lines 79-88):
```typescript
return wagmiSendTransaction.sendTransactionAsync({
    to: params.contractAddress ? params.contractAddress : addressAndAbi.address,
    data: dataWithReferral as `0x${string}`,
    value: params.value,
    gas: params.gas ? params.gas : null,
    gasLimit: params.gas ? params.gas : null,
    maxFeePerGas: 25.001e9,           // <- REMOVE: too low
    maxPriorityFeePerGas: 1e8,        // <- REMOVE: too low
    type: 2 
} as any);
```

New:
```typescript
const estimatedFees = await walletClient.estimateFeesPerGas();
return wagmiSendTransaction.sendTransactionAsync({
    to: params.contractAddress ? params.contractAddress : addressAndAbi.address,
    data: dataWithReferral as `0x${string}`,
    value: params.value,
    gas: params.gas ? params.gas : null,
    gasLimit: params.gas ? params.gas : null,
    ...estimatedFees,                  // <- ADD: dynamic fees
    type: 2 
} as any);
```

Alternative (simpler, let wagmi handle it):
```typescript
return wagmiSendTransaction.sendTransactionAsync({
    to: params.contractAddress ? params.contractAddress : addressAndAbi.address,
    data: dataWithReferral as `0x${string}`,
    value: params.value,
    gas: params.gas ? params.gas : null,
    gasLimit: params.gas ? params.gas : null,
    // Remove maxFeePerGas and maxPriorityFeePerGas - let wallet estimate
    type: 2 
} as any);
```

## Validation
- Transactions should use current network gas prices
- No more "transaction underpriced" or "replacement transaction underpriced" errors
- Transactions should be included in blocks in a reasonable time

## Notes
- The hardcoded values (25.001 Gwei max fee, 0.1 Gwei priority fee) were likely reasonable when set but are now obsolete
- Modern wallets (MetaMask) can estimate gas fees automatically when these parameters are omitted
- Using `walletClient.estimateFeesPerGas()` gives us programmatic control while keeping fees dynamic
